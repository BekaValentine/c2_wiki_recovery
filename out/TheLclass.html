In all seriousness, I'm with Jeff on this one, and we shouldn't blame him for the problems of Java language. He basically applied the principle ProgramIntoaLanguage, some may not agree to it and may prefer not to go againstb the grain of a language, but others (including me) take quite a pleasure into subverting a language [LanguageSubversionResponse]-- especially a language like Java.

Since a lot of languages makes it very convenient for programmers to express lists:

[x1,x2,x3, ... xn]
or (list x1 x2 ... xn)

versus

l= new LinkedList(); l.add(x1); ... l.add(xn) // note that in this example, he couldn't even bring himself to use the same number of elements!

he just corrected the language deficiency and now he can create lists as L.l(x1, ..., xn). With the new features in Java 1.5 this can even become l(x1,x2,...,xn). I used similar notations although I'm fond of using _ for this kind of trivial methods, and a very similar "trick the damn Java language" scheme is used in JScheme implementation. Most of the cases when programmers need to construct lists of fixed sizes from identifiers available in scope (as opposed to say iterating an algorithm and adding elements to a list), a maximum of 100 identifiers is more than enough.

So the bottom line is: who cares if Java doesn't support this or that, as long as there is a convenient way of faking it ? For example imbricated procedures can be faked quite convincingly by named inner classes inside the methods. Another example would be EmulateKeywordAndDefaultParameters. The only inconvenience that I still stumble on is CoRoutines. Yes you can emulate them with objects, but it ain't exactly as nice as emulating lists. --CostinCozianu
